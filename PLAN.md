## Product Summary


Map-first date journal for the actively-dating single person. Open to a map of every place you've been on a date, colored by rating. Tap a pin to read the journal entry. Log a new spot in 4 deliberate steps. Share your top-rated map as a native share sheet.


**Target user:** Someone actively dating who wants to never repeat a bad spot, track vibes across different dates, and discover genuinely good experiential places.


**Core loop:** Log → Map → Share

## Data Model


### users
```sql
id uuid PK
email text
name text
avatar_url text (nullable)
username text unique nullable -- set on first share action
profile_public bool default false
created_at timestamptz
```


### places (per-user, not shared)
```sql
id uuid PK
user_id uuid FK users
lat float8 NOT NULL
lng float8 NOT NULL
title text NOT NULL
type enum (scenic | activity | food | event | other)
cost enum (free | $ | $$ | $$$)
external_place_id text (nullable -- Mapbox Place ID)
created_at timestamptz
```


### visits
```sql
id uuid PK
user_id uuid FK users
place_id uuid FK places
rating     int2 CHECK (rating BETWEEN 1 AND 10)  -- materialized, recomputed from rank_order
rank_order float8                                 -- source of truth for comparison ordering (0.0–1.0)
with_person text (private, never public)
date_time timestamptz
duration_minutes int4 (nullable)
notes text CHECK (char_length(notes) <= 1500) (nullable)
photos text[] -- storage paths NOT signed URLs
created_at timestamptz
```
> **Migration needed:** `ALTER TABLE visits ADD COLUMN rank_order float8;`  
> Also add RPC: `log_visit_with_rank(visit_data, rank_order) → inserts visit + recomputes all user ratings`  
> In V1, this logic will live in-app as a local JavaScript function instead of a Postgres RPC.


### pins (wishlist / saved)
```sql
id uuid PK
user_id uuid FK users
place_id uuid FK places
created_at timestamptz
```


### public_profiles view
```sql
CREATE VIEW public_profiles AS
SELECT u.username, p.title, p.type, p.cost, v.rating, pl.lat, pl.lng
FROM visits v
JOIN places pl ON pl.id = v.place_id
JOIN users u ON u.id = v.user_id
WHERE u.profile_public = true
  AND v.rating >= 7;
```
> In V1, this view will be emulated as an in-memory query over the local SQLite dataset instead of a live Postgres view.


### RLS policy summary
- `users`: full CRUD where `id = local_uid()` (simulated)
- `places`: full CRUD where `user_id = local_uid()`
- `visits`: full CRUD where `user_id = local_uid()`
- `pins`: full CRUD where `user_id = local_uid()`
- `public_profiles` view: readable only to authenticated user in-app, no external client access (no live Supabase RLS)

## Phases

**Goal:** Running app that can simulate auth and render a blank Mapbox map, all hosted locally and testable in Xcode simulator.

Tasks:
1. `npx create-expo-app datespot --template expo-template-blank-typescript`

**Exit criteria:** `eas build --platform ios --profile development` passes; map renders in simulator; auth works as a local state simulation.

**Goal:** Logged visits appear as colored pins on the map, using only local data.

Tasks:
1. Query `visits JOIN places` for current user in local storage, return `{id, lat, lng, rating, type}`

8. Empty state: "Log your first spot" FAB centered on map


**Exit criteria:** Pins render with correct colors and icons; cluster tap zooms; filter bar updates visible pins; empty state FAB shows on fresh account.

### Phase 2 — Log Flow (4-step sheet)
**Goal:** User can log a visit in 4 steps from the FAB, storing everything locally.

Step 1 — Location:
- "Use my current location" (expo-location, request permission lazily)
- "Drop a pin" — modal map, long-press places draggable pin, Confirm → lat/lng
- "Search place" — free-text search over a local mock catalogue of places; no external Nominatim or MapKit calls.

Step 2 — Spot details:
- Title: free text
- Type: pill select (Scenic/Viewpoint, Activity, Food & Drink, Event, Other)
- Cost: pill select (Free, $, $$, $$$)

Step 3 — Journal:
- With whom: free text (private)
- Date: date picker (default today)
- Duration: picker (30m, 1hr, 1.5hr, 2hr, 2.5hr, 3hr, 3hr+)
- Notes: multiline text, char count shown, warn at 1,400 chars, hard block at 1,500
- Photos: up to 3 via expo-image-picker (lazy permission), JPEG/PNG, max 5MB
  - Inline error for oversized: "Photo too large — try a smaller image"
  - Upload failure toast: "Couldn't upload photo — your log was saved without it" — in this V1, this just means the photo was not persisted locally.

Step 4 — Rating:

**Goal:** Capture how good this spot was relative to all other spots this user has been on dates at, then store it as a 1–10 rating used for pin colors, filters, and sharing.

**Data model:**
- `visits.rank_order float8` — source of truth for ordering (0.0–1.0, fractional indexing)
- `visits.rating smallint` — materialized 1–10, recomputed from rank positions after every insert
- Recomputation is done in-app in JavaScript (same logic as `log_visit_with_rank`) instead of a live Postgres RPC.

```
rank_order: 0.0  0.2  0.35  0.5  0.8  1.0
rating:       1    3    5     6    8   10
```

**First spot (no existing visits):**
- Show 1–10 slider with anchor labels:
  - 1: "Never again"
  - 5: "Solid — would go back"
  - 10: "All-time favorite"
- Show note below: "Rough placement — your rating will update as you log more spots"
- Store `rank_order = slider_value / 10`, `rating = slider_value`

**Subsequent spots (existing visits present):**
- Intro: "How good was this spot compared to other places you've been on dates?"
- Show pairwise comparisons — "Which date spot was better?":
  - `[📍 Current spot — <title>]`
  - `[≈ About the same]`
  - `[📍 <Existing spot title>]`
- App chooses pivot via binary search (start at middle of sorted `rank_order` list, move bounds up/down based on answer):
  - New spot wins → `hi = mid`, recurse upper half
  - About the same → place at `existing.rank_order + ε`, end immediately
  - Existing spot wins → `lo = mid`, recurse lower half
- Stop after 7 comparisons or when `hi - lo <= 1` (insertion point is clear)
- Insert new `rank_order = (visits[lo].rank_order + visits[hi].rank_order) / 2`

**Rating interpolation (runs after every insert, covers all user's visits):**
```
rating = round(1 + (rank_index / (total - 1)) * 9)
guard: if total == 1, rating = 5
```
- All N visits' ratings updated in the in-app function (live ratings — pin colors reflect current relative ranking)
- Pin colors update on map after next local query

**On edit (re-rate):**
- Manual slider override (1–10), stored directly as rating; rank_order set to `(rating - 1) / 9`
- OR "Re-compare" — lightweight re-insertion: same binary search flow, re-runs full re-rating

**State machine for comparison flow (lives in `useComparisonRating` hook):**
```
idle → loadingVisits → comparing[lo, hi, mid, count] → complete → done
                                                      ↘ abandoned (user quits → no write)
```

**Error handling (critical — do not skip):**
- Draft the visit in AsyncStorage before the in-app write
- On in-app write failure: show retry toast, visit stays in draft state
- `interpolateRating(n, 1)` must guard against division by zero (total=1 → return 5)

**Known limit (V2 TODO):**
- After ~52 bisections between the same two adjacent rank_order values, float64 precision is exhausted. Fix: `rebalance(userId)` in-app function that evenly redistributes rank_order values across [1]. Not needed for typical usage.

Submit:
1. Write draft to AsyncStorage
2. Write `places` row (if new location) into local store
3. Call local `log_visit_with_rank` function — inserts visit + recomputes all ratings atomically
4. Clear draft from AsyncStorage
5. Navigate back to Map tab, map re-queries from local store, new pin animates in with updated colors

**Exit criteria:** Full 4-step flow completes; comparison converges in ≤7 steps; "about the same" ends immediately; visit appears on map with correct color; all N existing pins re-color correctly; write failure shows retry toast; `interpolateRating(n,1)` returns 5.

### Phase 3 — Pin Detail + Edit/Delete
**Goal:** Tap a pin → full journal entry; edit and delete supported, all local.

Tasks:
1. Bottom sheet: photos carousel (read from local paths at open time), rating badge, notes, type chip, cost chip, date, duration, "with whom" (private)
2. Edit: opens log flow pre-populated with existing visit data
3. Delete: confirmation prompt "Delete this spot? This can't be undone." → destructive delete of visit row (place row kept unless no other visits)
4. Signed URL generation is skipped; photos are read directly from local paths instead.


**Exit criteria:** Pin detail sheet renders all fields; edit round-trips correctly; delete prompts then removes pin from map.


***


### Phase 4 — Offline Queue
**Goal:** Visit logs survive a connectivity gap, even though there is no live server.

Tasks:
1. On submit without connectivity: serialize form data to AsyncStorage queue key `datespot:offline_queue`
2. Show persistent banner: "No connection — your log will be saved when you're back online." — under offline this just means persisted in-app.
3. On app start and on reconnect (AppState / connectivity listener): read queue, validate each entry (discard malformed), attempt sync — in this V1, “sync” means committing to local store.
4. On successful local sync: remove entry from queue, re-query map
5. If force-quit before reconnect: queue is lost (acceptable V1 trade-off, surfaced in UI copy)


**Exit criteria:** Submit in airplane mode → banner shown → app restart / reconnect → visit syncs to local store → pin appears on map.

### Phase 5 — Profile + Share Sheet
**Goal:** User can share their top-rated map with one tap, using only local data.

Tasks:
1. Profile tab: display name, avatar (from local auth state), username (edit inline), "Share my spots" toggle, settings
2. Username creation: prompted on first tap of "Share my spots" if not set; alphanumeric 3–20 chars; uniqueness error: "Username taken — try another" — uniqueness checked in-app over local users.
3. `profile_public = true/false` toggle → immediate effect on local `public_profiles` emulation
4. Share sheet generation:
   a. Query visits with rating ≥ 7 from local store
   b. Fit-to-bounds zoom on those pins
   c. Wait for `onDidFinishLoadingMap` callback
   d. Capture screenshot with react-native-view-shot
   e. Native share sheet: deep link `datespot://profile/[username]` + screenshot image — no live web endpoint; this is a mock URI.
5. Delete account: confirmation prompt → delete all visits, pins, places, user row in local store → sign out


**Exit criteria:** Toggle + username flow works; share sheet produces non-blank screenshot; delete account removes all local data.

### Phase 6 — Auth Polish + First Launch
**Goal:** First-time user lands on an empty, inviting map, all handled locally.

## Review Findings (CEO + Design + Eng)

### CEO Review — Verdict: SHIP IT (with adjustments)
*(No changes to CEO text, only the implication of a local-hosted V1.)*

**Cut:** Phase 4 (Offline Queue) — pre-emptive infrastructure for a behavior (frequent logging) you haven't proven yet. Every hour on offline sync is an hour not spent on distribution or learning whether anyone logs twice. Move to V2.
*(In V1, “offline” just means local storage; no external sync story.)*

**Add:** Push notification "remind me to log" triggered ~2 hours after a manually-armed "I'm on a date now" tap (or later: calendar heuristic). This directly tests the core ritual-logging premise and is more valuable than any current V1 feature.
*(This reminder can be implemented purely locally with `expo-notifications` and no cloud-based scheduler.)*

**Riskiest assumption:** "Logging is a ritual, not a task." Actively-dating singles are often emotionally raw post-date and moving fast. If they don't log, there's no map, and the product collapses. This is the thing to validate first.

**Distribution gap:** No channel strategy in the plan. Must be a deliberate bet before Phase 7. Top candidates: r/dating / r/Bumble subreddits, dating TikTok creators, App Store search for "date ideas."
*(Even with local-only back-end, these channels still apply.)*


### Design Review — Key gaps to close before coding
*(Same content; only mental context changes to “local-only” vs cloud.)*

**Must decide before coding:**
1. **How does "Save" work?** The segmented control shows "Visited" vs "Saved" but the log flow has no save path. Long-press on map? Bookmark icon on search results? This blocks the data model and pin visual treatment for saved (wishlist) spots.
2. **Step 3 is too heavy.** Combining with-whom + date + duration + notes + photos in one screen will cause abandonment. Recommended: make photos optional Step 4, push rating to Step 5, and define a "Done" escape hatch to skip remaining steps.
3. **Log flow dismiss behavior.** Can the user swipe down mid-flow? Does it save a draft or discard? Not defined.

**Missing empty states:**
- "Saved" tab with zero pins
- Filter combination returning zero results
- Share sheet when user has <1 spot rated 7+ (nothing to screenshot)
- Pin detail with no photos

**Missing error states:**
- GPS denied → Step 1 fallback
- Mapbox tiles fail to load (show inline banner per design doc, but Step 1 modal needs a state too) — in V1, this can mean “loading Apple Maps style failed”
- Search autocomplete returns zero results
- Share screenshot generation failure

**Accessibility gap:** Pin color is the sole rating differentiator — fails for deuteranopia/protanopia. Must pair color with shape/pattern (e.g., circle size or icon badge). Rating slider needs `accessibilityValue`. Photos need `accessibilityLabel`.


### Eng Review — Technical risks to address
*(Same content, but interpreting “live Supabase” references as local-only.)

**Critical before Phase 0:**
- `@rnmapbox/maps` requires a **custom dev client** (`expo-dev-client`) — does NOT work in Expo Go. Confirm day one. Also requires two tokens: `MAPBOX_DOWNLOADS_TOKEN` (secret, for SDK install) and `MAPBOX_TOKEN` (public, for tiles). These are separate.
- Android needs `MAPBOX_DOWNLOADS_TOKEN` in `~/.gradle/gradle.properties` or EAS secrets.
- iOS needs `MBXAccessToken` in `Info.plist` via config plugin — missing gives silent blank map.

**Add to tech stack:** `@tanstack/react-query` — needed for signed URL expiry handling (1-hour TTL), optimistic updates for offline queue, and background refetch. Without it you'll hand-roll cache invalidation across every screen.
*(In V1, use it for local-only caching/optimistic updates, not for live Supabase sockets.)*

**Clustering:** Must use Mapbox's native `clusterMaxZoom` / `clusterRadius` props on `ShapeSource`, NOT JS-side bounding box computation. JS approach is racey with map movement.

**Screenshot:** `onDidFinishLoadingMap` is NOT enough. Sequence must be: `fitBounds` → await `onMapIdle` (camera-settled) → `captureRef`. Also: pass the MapView's native ref, not a wrapper View ref.

**Photo upload:** `expo-image-picker` returns a `file://` URI. Must `fetch(uri).then(r => r.blob())` before using / copying locally — do not pass URI string directly. On Android, blob construction may fail; fall back to `FileSystem.readAsStringAsync` with base64.

**Supabase Storage RLS gap:** — *skip discussion of Supabase Storage RLS*; in V1, all photos are stored and read locally, so no external storage-policy rules apply.

**Offline queue:** Validate schema shape (not just JSON parse) to catch partial writes. Add `AppState active` listener as secondary sync trigger — `NetInfo` reconnect events are unreliable on iOS in background.
*(In V1, “sync” means replaying to local store, not to a live server.)*
